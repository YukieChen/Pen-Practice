# 字帖產生器「自選字型、自行產生字帖」公開風險

> 情境：程式公開給一般大眾使用，使用者自己選擇字型檔、自己產生字帖。  
> 本文整理此情境下的風險與已/未防護項目。

---

## 一、風險總覽

```
                    使用者上傳字型檔
                            │
                            ▼
    ┌───────────────────────────────────────────────────────────┐
    │ 選檔 (UI) → 大小檢查 → 讀入記憶體 (arrayBuffer) → 解析 (opentype) │
    └───────────────────────────────────────────────────────────┘
         │              │                    │                    │
         ▼              ▼                    ▼                    ▼
    副檔名/MIME     【已修】20MB         記憶體/DoS         惡意/畸形檔
    僅提示可繞過    超過即拒絕          大檔=卡死           解析器風險
```

---

## 二、風險清單與對策

| # | 風險 | 說明 | 目前狀態 | 建議 |
|---|------|------|----------|------|
| 1 | **字型檔超過 20MB** | 規格要求 < 20MB；未檢查時大檔會整份讀入記憶體，造成分頁卡死或 OOM。 | ✅ **已修**：上傳前檢查 `file.size`，超過則顯示「字體檔案大小超過 20MB 限制，請選擇較小的字體檔」並清除選擇。 | 維持；可考慮在 UI 文案註明「建議 20MB 以內」。 |
| 2 | **檔案類型只靠副檔名** | `accept=".ttf,.otf"` 僅為提示，使用者可選任意檔（如 .exe 改名 .ttf）。 | ⚠️ 依賴 `opentype.parse()` 拋錯表示「非字型」；非字型二進位可能造成解析時當掉或耗時。 | 可選：在讀取後用 magic bytes 檢查（TTF: `0x00010000` 或 `true`，OTF: `OTTO`），再送解析。 |
| 3 | **惡意或畸形字型檔** | 刻意構造的 TTF/OTF（畸形 table、超大 glyph 數、超深結構）可能觸發解析器 bug，造成長時間迴圈或記憶體暴增。 | ⚠️ 全在瀏覽器端，無 RCE；風險為單一分頁凍結/崩潰（影響該使用者體驗）。 | 依賴 opentype.js 更新；可考慮解析逾時或 Worker 隔離（實作成本較高）。 |
| 4 | **記憶體與 DoS** | 大檔（<20MB 但接近 20MB）整份 `arrayBuffer()` 進記憶體，低階裝置或多分頁時易卡頓。 | ✅ 20MB 上限已限制規模；解析時間規格為約 0.5–2 秒（見 02_technical-feasibility）。 | 可選：顯示「解析中…」避免使用者重複點擊。 |
| 5 | **檔名 / 字型名稱顯示** | 若將來顯示字型內部 name table，未跳脫可能 XSS。 | ✅ 目前僅顯示 `file.name`（檔名），未從字型讀取並渲染 name table 字串。 | 若日後顯示字型名稱，一律用 `textContent` 或跳脫。 |
| 6 | **JSON 設定檔** | 上傳 JSON 有 5MB 限制與格式驗證，與字型上傳分開。 | ✅ 已有 `MAX_JSON_BYTES` 與錯誤訊息。 | 無需變更。 |

## 三、資料流與檢查點（目前實作）

```
  [使用者選檔]
        │
        ▼
  app.ts 字型 callback
        │
        ├─ file === null → 清除、refresh
        ├─ file.size > MAX_FONT_BYTES (20MB) → 設 error、清 customFontFile、return  ← 新增
        └─ 否則 → loadCustomFont(file)
                    │
                    ▼
              font-manager.loadCustomFont
                    │
                    ├─ file.arrayBuffer()  ← 整份進記憶體
                    └─ opentype.parse(buffer)  ← 可能拋錯或耗時
```

---

## 四、規格對齊

- **E03（03_scenarios / 06 E03）**：上傳超過 20MB 字體檔 → 上傳前檢查、顯示錯誤訊息、不開始解析。  
- **實作**：`MAX_FONT_BYTES` 於 `src/utils/validators.ts`，於 `src/ui/app.ts` 字型 callback 內檢查，錯誤訊息與規格一致。

---

## 五、小結

- **已處理**：字型檔 20MB 上限（你發現的缺口）已補上，符合安全要求與規格。
- **其餘風險**：多為「使用者自己選檔、自己解析」在瀏覽器內的體驗與穩定性問題（大檔/畸形檔導致卡頓或當掉），而非對伺服器或他人的直接攻擊；目前架構下無後端上傳，風險範圍限於該使用者的分頁。
- **若要再強化**：可加 magic bytes 檢查、解析中提示或（進階）Worker + 逾時，依需求與成本取捨。
