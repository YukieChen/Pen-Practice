## 1. Happy Path 情境

### S01: 無說明列模式 — 基本產生

```
Scenario: 使用者以預設字體產生無說明列字帖

  Given 使用者開啟字帖產生器網頁
    And 預設模式為「不需要說明列」
    And 預設字體為內建書寫用楷體

  When 使用者在第 1 段練習字欄位輸入 "寅宙守宿"
    And 使用者在第 2 段練習字欄位輸入 "至盆盤鹽"
    And 使用者在第 3 段練習字欄位輸入 "勵勸即勤"
    And 使用者在第 4 段練習字欄位輸入 "績蠟讀曦"

  Then 統計顯示 "共 4 段, 16 字 │ 無說明: 1 頁"
    And 「產生字帖 PDF」按鈕變為可點擊狀態
    And 說明文字輸入欄不顯示

  When 使用者點擊「產生字帖 PDF」

  Then 按鈕顯示產生進度
    And 系統產生 1 頁 A4 PDF
    And PDF 包含 72x48 格線 (3 層: fine/cell/section)
    And PDF 包含 4 段 x 4 字 = 16 個字的 glyph path
    And 每段第 1 行第 1 欄為範字 (opacity 100%)
    And 每段欄 1-2 為引導字 (opacity 25%)
    And 每段欄 3-4 為空白格線
    And 每段有 6 行練習列
    And 瀏覽器自動下載 PDF 檔案
    And 檔案名稱格式為 "鋼筆字帖_YYYYMMDD_HHmmss.pdf"
```

### S02: 有說明列模式 — 基本產生

```
Scenario: 使用者以預設字體產生有說明列字帖

  Given 使用者開啟字帖產生器網頁

  When 使用者選擇「需要說明列」模式

  Then 每段的說明文字輸入欄以動畫展開

  When 使用者在第 1 段輸入說明 "天覆者凡畫皆冒於其下" 及練習字 "寅宙守宿"
    And 使用者在第 2 段輸入說明 "地載者有畫皆托於其上" 及練習字 "至盆盤鹽"
    And 使用者在第 3 段輸入練習字 "勵勸即勤" (說明留空)
    And 使用者在第 4 段輸入練習字 "績蠟讀曦" (說明留空)

  Then 統計顯示 "共 4 段, 16 字 │ 有說明: 1 頁"

  When 使用者點擊「產生字帖 PDF」

  Then 系統產生 1 頁 A4 PDF
    And 每段第 1 行為說明列 (跨全寬, 高 3 fine-rows)
    And 第 1 段說明列文字為 "天覆者凡畫皆冒於其下"
    And 第 2 段說明列文字為 "地載者有畫皆托於其上"
    And 第 3 段說明列為空白 (無文字)
    And 第 4 段說明列為空白 (無文字)
    And 每段有 5 行練習列 (扣除說明列)
    And 瀏覽器自動下載 PDF 檔案
```

### S03: 自訂字體上傳

```
Scenario: 使用者上傳自訂字體檔並產生字帖

  Given 使用者開啟字帖產生器網頁
    And 使用者已輸入練習字 "永"

  When 使用者在字體選擇區點擊上傳區域
    And 使用者選擇一個合法的 .ttf 字體檔 (如 "微軟正黑體.ttf", 15MB)

  Then 上傳區顯示進度條
    And 進度條完成後顯示 "微軟正黑體.ttf (15MB)"
    And 字體預覽區顯示「永」字 (使用上傳字體的 glyph path 渲染)
    And 「使用上傳字體」選項自動選中

  When 使用者點擊「產生字帖 PDF」

  Then 系統使用上傳字體的 glyph path 產生 PDF
    And 字帖中的「永」字筆劃精準填滿 3x3 字格 (90% fill ratio)
    And 不因微軟正黑體的 metrics 差異而溢出或偏移
```

### S04: 多頁產生 (無說明列)

```
Scenario: 無說明列模式下字數超過 16 字，自動分頁

  Given 使用者開啟字帖產生器網頁
    And 模式為「不需要說明列」

  When 使用者在第 1 段輸入 10 字
    And 使用者在第 2 段輸入 10 字

  Then 統計顯示 "共 2 段, 20 字 │ 無說明: 2 頁"

  When 使用者點擊「產生字帖 PDF」

  Then 系統合併所有段落字池為扁平清單 (20 字)
    And 系統產生 2 頁 PDF
    And 第 1 頁包含前 16 字 (4段x4字)
    And 第 2 頁包含後 4 字 (第1段4字, 其餘段空白)
    And 第 2 頁空白段僅顯示格線 (頁面格線畫滿)
    And 兩頁格線結構完全相同
```

### S05: 不足 16 字的處理 — 頁面仍畫滿

```
Scenario: 使用者輸入少於 16 字，格線仍畫滿一頁

  Given 使用者開啟字帖產生器網頁
    And 模式為「不需要說明列」

  When 使用者在第 1 段輸入 "永遠幸福快樂" (6 字)

  Then 統計顯示 "共 1 段, 6 字 │ 無說明: 1 頁"

  When 使用者點擊「產生字帖 PDF」

  Then 系統產生 1 頁 PDF
    And 第 1 段包含 4 字: 永 遠 幸 福
    And 第 2 段包含 2 字: 快 樂 (字組C, D 空白)
    And 第 3 段完全空白 (僅格線)
    And 第 4 段完全空白 (僅格線)
    And 整頁格線完整繪製 (72x48 fine grid)
```

### S06: 僅輸入 1 字

```
Scenario: 使用者僅輸入 1 個字，格線仍畫滿一頁

  Given 使用者開啟字帖產生器網頁

  When 使用者在第 1 段輸入 "永"

  When 使用者點擊「產生字帖 PDF」

  Then 系統產生 1 頁 PDF
    And 第 1 段字組 A 有「永」字 (範字+引導字)
    And 第 1 段字組 B, C, D 為空白格線
    And 第 2-4 段完全空白 (僅格線)
    And 整頁格線完整繪製 (供使用者自行運用空白區)
```

### S07: 拖曳上傳字體

```
Scenario: 使用者以拖曳方式上傳字體

  Given 使用者開啟字帖產生器網頁

  When 使用者將 .otf 檔案拖曳到字體上傳區域

  Then 上傳區域高亮顯示 (視覺回饋)
    And 放開後開始解析字體
    And 解析完成後顯示字體名稱與檔案大小
    And 字體預覽區更新
```

### S08: 有說明列 — 字池超過 4 字隨機挑選

```
Scenario: 有說明列模式下，段落字池超過 4 字，隨機挑 4 字

  Given 使用者選擇「需要說明列」模式

  When 使用者在第 1 段輸入說明 "天覆者凡畫皆冒於其下"
    And 使用者在第 1 段輸入練習字 "寅宙守宿寶家窩宰" (8 字)

  Then 第 1 段顯示 "共 8 字，有說明: 隨機挑 4 字"

  When 使用者點擊「產生字帖 PDF」

  Then 系統從 8 字中隨機挑選 4 字
    And 產生的 PDF 第 1 段僅包含被挑選的 4 字
    And 第 1 段說明列顯示 "天覆者凡畫皆冒於其下"
    And 每次產生可能挑到不同的 4 字
```

### S09: 無說明列 — 字池超過 4 字全部印出

```
Scenario: 無說明列模式下，段落字池全部印出

  Given 使用者選擇「不需要說明列」模式

  When 使用者在第 1 段輸入 "寅宙守宿寶家窩宰" (8 字)
    And 使用者在第 2 段輸入 "至盆盤鹽" (4 字)

  Then 統計顯示 "共 2 段, 12 字 │ 無說明: 1 頁"

  When 使用者點擊「產生字帖 PDF」

  Then 系統合併所有字為扁平清單 (12 字)
    And 產生 1 頁 PDF
    And 第 1 段: 寅宙守宿
    And 第 2 段: 寶家窩宰
    And 第 3 段: 至盆盤鹽
    And 第 4 段: 空白 (僅格線)
```

### S10: 上傳 JSON 設定檔

```
Scenario: 使用者上傳 JSON 設定檔自動填入段落

  Given 使用者開啟字帖產生器網頁

  When 使用者切換到「上傳設定檔」
    And 使用者上傳以下 JSON 檔案:
      {
        "sections": [
          { "description": "01、天覆者...", "characters": ["寅","宙","守","宿"] },
          { "description": "02、地載者...", "characters": ["至","盆","盤","鹽"] }
        ]
      }

  Then 系統解析 JSON 成功
    And 第 1 段說明自動填入 "01、天覆者..."
    And 第 1 段練習字自動填入 "寅宙守宿"
    And 第 2 段說明自動填入 "02、地載者..."
    And 第 2 段練習字自動填入 "至盆盤鹽"
    And 使用者仍可手動修改各欄位
```

### S11: 下載設定檔

```
Scenario: 使用者將目前輸入內容下載為 JSON 設定檔

  Given 使用者已在第 1 段輸入說明 + 練習字 (含 >4 字)
    And 使用者已在第 2 段輸入說明 + 練習字

  When 使用者點擊「下載設定檔」

  Then 瀏覽器下載 JSON 檔案
    And 檔案名稱格式為 "字帖設定_YYYYMMDD_HHmmss.json"
    And JSON 包含完整的 sections 陣列
    And 每段的 characters 包含所有字 (不因 >4 字而截斷)
```

### S12: 選擇系統預設設定檔 — 黃自元間架結構九十二法

```
Scenario: 使用者選擇系統預設批次並產生字帖

  Given 使用者開啟字帖產生器網頁
    And 使用者已上傳自訂字體

  When 使用者切換到「系統預設」
    And 使用者點擊「第 1-16 法」批次按鈕

  Then 系統自動填入 16 段預設內容 (第 1-16 法)
    And 每段包含書法結構說明 + 練習字池
    And 統計顯示 "共 16 段 │ 有說明: 4 頁"

  When 使用者選擇「需要說明列」模式
    And 使用者點擊「產生字帖 PDF」

  Then 系統產生 4 頁 PDF
    And 使用自訂字體渲染
    And 若自訂字體缺字，對應格子留白僅顯示格線，並於 UI 顯示缺字警告
    And 每段字池 >4 字則隨機挑 4 字
    And 每段包含說明列
```

### S13: 無說明模式 + 設定檔有說明 → 忽略說明

```
Scenario: 無說明模式下上傳含說明的設定檔，忽略說明

  Given 使用者選擇「不需要說明列」模式

  When 使用者上傳含說明的 JSON 設定檔:
    {
      "sections": [
        { "description": "天覆者...", "characters": ["寅","宙","守","宿","寶","家","窩","宰"] },
        { "description": "地載者...", "characters": ["至","盆","盤","鹽"] }
      ]
    }

  Then 段落自動填入
    And 說明欄位被填入但不顯示 (因為無說明模式)

  When 使用者點擊「產生字帖 PDF」

  Then 系統忽略所有說明文字
    And 合併所有字池: 12 字全部印出
    And 產生 1 頁 PDF (無說明列)
    And 12 字依序分配到 3 段 (第4段空白)
```

### S14: 缺字留白格 + 提示

```
Scenario: 上傳字體缺少部分字元，留白格顯示並提示缺字

  Given 使用者上傳了一個字體檔
    And 字體成功解析

  When 使用者輸入練習字 "永遠曦"
    And 字體包含「永」「遠」但缺少「曦」

  Then 顯示警告訊息 "此字體缺少以下字元: 「曦」，這些字元在字帖中將顯示為空白格"
    And 產生按鈕仍可點擊

  When 使用者點擊「產生字帖 PDF」

  Then PDF 中「永」「遠」使用上傳字體的 glyph path 渲染
    And PDF 中與「曦」對應的所有格子僅顯示 3x3 格線，無任何筆劃 (留白練習格)
    And 其他字的大小與排列仍依 Ink BBox 縮放正常呈現
```

## 2. Error Path 情境

### E01: 空白輸入 — 按鈕禁用

```
Scenario: 使用者未輸入任何字

  Given 使用者開啟字帖產生器網頁
    And 所有段落的練習字欄位為空

  Then 「產生字帖 PDF」按鈕為禁用狀態 (灰色)
    And 「下載設定檔」按鈕為禁用狀態
    And 按鈕 hover 時顯示 tooltip "請先輸入至少一個中文字"

  When 使用者在練習字欄位輸入空格或英文字母 "abc 123"

  Then 輸入被自動過濾，欄位顯示為空
    And 按鈕仍為禁用狀態
```

### E02: 無效字體檔案

```
Scenario: 使用者上傳非字體檔案

  Given 使用者開啟字帖產生器網頁

  When 使用者上傳一個 .png 檔案 (偽裝為字體)

  Then 系統嘗試用 opentype.js 解析
    And 解析失敗
    And 顯示錯誤訊息 "無法解析此字體檔案，請確認為 .ttf 或 .otf 格式"
    And 字體選擇保持為「預設字體」
    And 上傳區域顯示「重新上傳」按鈕
```

### E03: 字體檔案過大

```
Scenario: 使用者上傳超過 20MB 的字體檔

  Given 使用者開啟字帖產生器網頁

  When 使用者選擇一個 25MB 的字體檔

  Then 系統在上傳前檢查檔案大小
    And 顯示錯誤訊息 "字體檔案大小超過 20MB 限制，請選擇較小的字體檔"
    And 不開始解析
```

### E04: JSON 設定檔格式錯誤

```
Scenario: 使用者上傳格式不正確的 JSON 設定檔

  Given 使用者開啟字帖產生器網頁

  When 使用者上傳一個格式不合法的 JSON 檔案 (缺少 sections 欄位或非陣列)

  Then 顯示錯誤訊息 "設定檔格式不正確，請確認為合法的 JSON 格式且包含 sections 陣列"
    And 段落內容不受影響
```

### E05: 說明文字過長

```
Scenario: 使用者輸入超長說明文字

  Given 使用者選擇「需要說明列」模式
    And 使用者已輸入練習字

  When 使用者在第 1 段說明輸入超過 50 字的文字

  Then 輸入框顯示字數警告 "說明文字最長 50 字"
    And 超過 50 字的部分被截斷
```

### E06: 模式切換時保留輸入

```
Scenario: 使用者切換模式後不遺失已輸入內容

  Given 使用者選擇「需要說明列」模式
    And 使用者已在第 1 段輸入說明 "橫平豎直" 及練習字 "永遠幸福"

  When 使用者切換為「不需要說明列」模式

  Then 說明欄位以動畫收合 (隱藏但保留值)
    And 練習字 "永遠幸福" 保持不變

  When 使用者再次切換為「需要說明列」模式

  Then 說明欄位展開
    And 第 1 段說明仍顯示 "橫平豎直" (未遺失)
```

## 3. Edge Case 情境

### EC01: 輸入含混合字元

```
Scenario: 使用者輸入含非中文字元的混合文字

  Given 使用者開啟字帖產生器網頁

  When 使用者在第 1 段練習字輸入 "Hello 永遠！ 幸福123"

  Then 系統自動過濾非中文字元
    And 實際練習字為 "永遠幸福" (4 字)
```

### EC02: 重複字元

```
Scenario: 使用者輸入重複的字

  Given 使用者開啟字帖產生器網頁

  When 使用者在第 1 段輸入 "永永永永"

  Then 系統接受重複字元 (不去重)
    And 第 1 段顯示 4 字
```

### EC03: 瀏覽器不支援

```
Scenario: 使用者使用過舊的瀏覽器

  Given 使用者使用 IE 11 開啟網頁

  Then 頁面顯示兼容性提示
    And 提示內容: "請使用 Chrome、Firefox、Safari 或 Edge 最新版本"
    And 產生功能不可用
```

### EC04: PDF 產生過程中的錯誤

```
Scenario: PDF 產生過程中發生非預期錯誤

  Given 使用者已填寫所有必要資訊

  When 使用者點擊「產生字帖 PDF」
    And PDF 產生過程中發生錯誤 (如記憶體不足)

  Then 進度條停止
    And 顯示錯誤訊息 "產生失敗，請重新嘗試"
    And 按鈕恢復為可點擊狀態
    And 使用者已輸入的內容不受影響
```

### EC05: 有說明模式多頁分頁

```
Scenario: 有說明模式下超過 4 段自動分頁

  Given 使用者選擇「需要說明列」模式

  When 使用者新增並填寫 6 段內容 (每段各 4 字)

  Then 統計顯示 "共 6 段, 24 字 │ 有說明: 2 頁"

  When 使用者點擊「產生字帖 PDF」

  Then 系統產生 2 頁 PDF
    And 第 1 頁: 4 段 (各含說明列 + 4 字)
    And 第 2 頁: 2 段 (含說明列 + 4 字) + 2 段空白 (僅格線)
```

### EC06: 設定檔上傳後再手動修改

```
Scenario: 上傳設定檔後使用者手動修改內容

  Given 使用者上傳了一個 JSON 設定檔
    And 段落已自動填入

  When 使用者手動修改第 1 段的練習字 (增加 2 字)
    And 使用者手動修改第 2 段的說明文字

  Then 修改即時生效
    And 統計即時更新
    And 使用者可正常產生 PDF

  When 使用者點擊「下載設定檔」

  Then 下載的 JSON 包含修改後的內容 (非原始上傳內容)
```
